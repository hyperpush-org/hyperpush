# todo-postgres

This project was generated by `meshc init --template todo-api --db postgres`.

It keeps the clustered-function contract source-first and route-free while moving the starter onto the serious PostgreSQL path: `DATABASE_URL` configuration, pool-backed startup, migration-owned schema creation, helper-backed CRUD, and a Dockerfile that packages the binary produced by `meshc build .`.

## Clustered contract

- `mesh.toml` stays package-only and intentionally omits `[cluster]`
- `main.mpl` validates config, opens the local PostgreSQL pool, then boots through `Node.start_from_env()` before starting the local HTTP server
- `work.mpl` declares `@cluster pub fn sync_todos()`
- the runtime-owned handler name is derived from the ordinary source function name as `Work.sync_todos`
- `GET /todos` and `GET /todos/:id` use `HTTP.clustered(1, ...)` so the starter truthfully dogfoods the shipped wrapper while `GET /health` and mutating routes stay local

## HTTP routes

- `GET /health` ŌĆö local runtime + PostgreSQL/starter configuration snapshot without leaking `DATABASE_URL`
- `GET /todos` ŌĆö list todos through `HTTP.clustered(1, ...)`
- `GET /todos/:id` ŌĆö fetch one todo through `HTTP.clustered(1, ...)`
- `POST /todos` ŌĆö create from `{"title":"Buy milk"}`
- `PUT /todos/:id` ŌĆö toggle completion
- `DELETE /todos/:id` ŌĆö delete a todo

Mutating routes (`POST`, `PUT`, `DELETE`) stay local and go through the actor-backed limiter in `services/rate_limiter.mpl`.

## Schema and migrations

`migrations/` owns schema creation. Apply migrations before booting the API:

```bash
meshc migrate . status
meshc migrate . up
```

The runtime opens the pool and serves HTTP only after configuration succeeds. It does not create tables at startup.

## Environment

- `DATABASE_URL` ŌĆö required PostgreSQL connection string
- `PORT` ŌĆö HTTP port (`8080` by default)
- `TODO_RATE_LIMIT_WINDOW_SECONDS` ŌĆö limiter window length (`60` by default)
- `TODO_RATE_LIMIT_MAX_REQUESTS` ŌĆö allowed mutating requests per window (`5` by default)
- `.env.example` ŌĆö starter copy of the same local-development keys
- `MESH_CLUSTER_COOKIE`, `MESH_NODE_NAME`, `MESH_DISCOVERY_SEED`, `MESH_CLUSTER_PORT`, `MESH_CONTINUITY_ROLE`, `MESH_CONTINUITY_PROMOTION_EPOCH` ŌĆö optional clustered-runtime env when you want the runtime-owned operator flow

## Local run

```bash
meshc migrate . up
meshc build .
DATABASE_URL=postgres://postgres:postgres@127.0.0.1:5432/todo-postgres \
PORT=8080 \
TODO_RATE_LIMIT_WINDOW_SECONDS=60 \
TODO_RATE_LIMIT_MAX_REQUESTS=5 \
./todo-postgres
```

## Runtime inspection

When you boot the app in clustered mode, inspect the runtime-owned state through Mesh CLI surfaces instead of package-owned control routes:

```bash
meshc cluster status <node-name@host:port> --json
meshc cluster continuity <node-name@host:port> --json
meshc cluster continuity <node-name@host:port> <request-key> --json
meshc cluster diagnostics <node-name@host:port> --json
```

Use the continuity list form first to discover runtime-owned startup records.

## Docker

Build the Linux binary you want to ship, then package that `./output` artifact into the image:

```bash
meshc build .
docker build -t todo-postgres .
```

The container expects a migrated database and a runtime `DATABASE_URL`; it does not run migrations or create schema at startup. Re-run `meshc build .` before rebuilding the image, and build on the Linux target you plan to run because the Dockerfile copies the already-compiled `./output` binary into the container.

```bash
docker run --rm \
  -p 8080:8080 \
  -e PORT=8080 \
  -e DATABASE_URL=postgres://postgres:postgres@host.docker.internal:5432/todo-postgres \
  todo-postgres
```
